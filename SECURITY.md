# Security Policy

We take the security of Wants AI and its users seriously. Thank you for
helping us keep the project safe.

## Reporting a Vulnerability

**Please do not open public GitHub issues, discussions, or pull requests
for security vulnerabilities.** Public disclosure before a fix is
available puts users at risk.

You have two private reporting channels — pick whichever you prefer:

### 1. GitHub Security Advisories (preferred)

Open a private vulnerability report directly from the repository:

> **[Report a vulnerability via GitHub](https://github.com/wants-chat/wants-chat/security/advisories/new)**

(Or navigate to the **Security** tab → **Advisories** → **Report a
vulnerability**.)

This creates a private draft advisory that only repository maintainers
can see. You can attach files, exchange messages with the team in
threaded comments, and we can later turn the advisory into a
coordinated public disclosure with a CVE.

### 2. Email

If you can't or don't want to use GitHub, email us at:

> **support@infoinlet.com** — please prefix the subject line with
> `[SECURITY] wants-chat:` so we can route it correctly.

### What to include

Whichever channel you use, please include as much of the following as
you can:

- A clear description of the vulnerability
- The exact version, branch, or commit hash you tested against
- Step-by-step instructions to reproduce
- A proof-of-concept (PoC) if you have one
- The potential impact (what an attacker can read, modify, or do)
- Any suggested fix or mitigation

The more we can reproduce on first read, the faster we can ship a fix.

## Scope

**In scope:**

- The `wants-chat` repository itself (frontend, backend, browser
  extension, app-builder, app-creator)
- Default configurations shipped in this repo (Docker Compose,
  `.env.example`, GitHub Actions workflows)
- Direct dependencies, where the vulnerability is exploitable in
  *our* usage of them

**Out of scope:**

- Apps generated by `app-creator/` from end-user prompts (the
  generated app is the user's responsibility, not ours, unless the
  vulnerability is in our generator templates)
- The hosted commercial instance at `wants.chat` (please report
  hosted-service issues to **support@infoinlet.com** with subject
  prefix `[SECURITY] wants.chat`)
- Third-party services we integrate with (OpenRouter,
  Cloudflare R2, AWS SES, etc.) — report those upstream
- Social engineering, physical attacks, and attacks requiring
  privileged local access to a victim's machine
- Self-XSS that requires the victim to paste arbitrary code into
  their browser devtools
- Missing security headers or best-practice hardening that doesn't
  enable a concrete attack
- Rate-limiting weaknesses, unless they bypass an existing protection
- Vulnerabilities in old, unsupported versions (see *Supported
  Versions* below)

## What Qualifies as a Security Issue

We consider the following high-priority:

- Authentication or authorization bypasses
- SQL injection, XSS, CSRF, SSRF, command injection, or other
  injection attacks
- Insecure direct object references (IDOR) and tenant isolation
  failures
- Remote code execution
- Privilege escalation
- Exposure of sensitive user data, secrets, or credentials
- Insecure default configurations that lead to data exposure
- Cryptographic weaknesses (broken hashing, predictable tokens, etc.)
- Vulnerable dependencies that are exploitable in our usage

If you're not sure whether something qualifies, **report it anyway**
and we'll triage it.

## Severity & Response Targets

We're a small team and respond on a best-effort basis. The targets
below are aspirational, not contractual:

| Severity | Example | Acknowledgment | Fix target |
|----------|---------|----------------|------------|
| **Critical** | RCE, auth bypass, mass data exposure | 2 business days | 7 days |
| **High** | IDOR, stored XSS, privilege escalation | 3 business days | 30 days |
| **Medium** | Reflected XSS, CSRF on non-critical actions | 5 business days | 60 days |
| **Low** | Information disclosure with minimal impact | 5 business days | Next regular release |

If we can't meet a target we'll keep you updated in the advisory
thread or by email.

## Coordinated Disclosure

Once a fix is ready, we will:

1. Cut a patched release (or merge to `main` if pre-1.0).
2. Publish a GitHub Security Advisory describing the issue, affected
   versions, the fix, and mitigation steps.
3. Request a CVE through GitHub's CVE Numbering Authority (CNA), if
   the issue warrants one.
4. Credit you in the advisory by name or handle, **with your
   permission**. If you'd rather stay anonymous, just say so.

We ask that you give us a reasonable window (typically up to 90 days,
sooner for low-risk issues) to ship a fix before publicly disclosing
the vulnerability. We'll work with you on the exact timeline.

## Supported Versions

Wants AI is pre-1.0. Until we cut a stable release, security fixes
are applied to:

| Version | Supported |
|---------|-----------|
| `main` branch | ✅ Yes |
| Tagged pre-releases | Best effort |
| Forks | Not by us — patch your fork from `main` |

After 1.0, we expect to support the latest minor release and the one
immediately preceding it.

## Safe Harbor

We won't pursue legal action against researchers who:

- Make a good-faith effort to comply with this policy
- Avoid privacy violations, data destruction, and service disruption
- Give us reasonable time to investigate and fix the issue before
  any public disclosure
- Only interact with accounts and data that they own, or that the
  account holder has given explicit permission to test

This is not a formal bug-bounty program — we don't currently offer
monetary rewards — but we will publicly credit researchers who help
us improve the project, and we're happy to provide a written
acknowledgment of your contribution on request.

## Questions

For anything security-related that isn't a vulnerability report
(policy questions, scope clarifications, dependency concerns), email
**support@infoinlet.com** or open a regular GitHub issue.
